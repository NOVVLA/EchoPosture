# Rules of Engagement

本文件规定 EchoPosture 项目的编辑、提交、回滚和分支规则。后来维护者必须先读本文件，再改仓库。

## 基本原则

- 不允许“想提交就提交、想不提交就不提交”。是否提交由改动风险和可观察性决定。
- 不得随意创建分支。默认在当前 `main` 上工作；只有用户明确要求、远端协作必须隔离、或需要保留长期并行方案时，才可创建分支。
- 不得用 `git reset --hard`、`git checkout -- <path>`、强推、删除 release、删除备份等破坏性动作来省事。确需执行时，必须先得到明确批准。
- 不得覆盖、回退或整理用户未要求处理的改动。发现非本人改动时，先识别影响范围，再决定是否绕开或配合。

## 必须提交的情况

满足任一条件时，改动完成并验证后必须形成 Git 提交：

- 直接可观察的改动：用户能看到、运行到、下载到、点击到、读到，或 release 附件会包含的变化。
- 修正错误改动：恢复误改文件、修复错误文档、修正错误 release 附件或撤销错误行为。
- 可能造成死锁、卡死、无法退出、无法清理覆盖层、摄像头释放失败、后台进程悬挂、UI 阻塞、构建卡住、发布流程卡住的变化。
- 改动影响启动器、托盘运行时、视觉覆盖层、GPU 模糊宿主、摄像头读取、发布包、release、备份或恢复流程。
- 单个文件内改动内容较大，或同一任务跨多个文件、多个模块、多个运行入口。
- 文档规则、冻结文件规则、发布规则、回滚规则发生变化。
- 用户明确要求“提交”、“推送”、“发布”或“恢复到 GitHub”。

## 可以不提交的情况

以下情况可以不提交，但必须保持工作树干净或明确说明遗留状态：

- 只读检查、日志查看、状态查询、release 查询、hash 查询。
- 临时预览服务、临时截图、临时 pid 文件等验证产物；这些应清理，不应提交。
- 未形成文件改动的解释、规划或代码阅读。
- 用户明确要求只分析、不改文件、不提交。

## 提交流程

每次提交前必须完成：

1. 运行 `git status --short`，确认只包含本任务相关文件。
2. 查看 `git diff` 或 `git diff --cached`，确认没有误改冻结文件、发布包、备份、运行时或生成物。
3. 按改动范围运行最小必要验证；构建、发布、UI、文档、Python、原生代码各自选择对应验证。
4. 提交信息必须说明真实意图，不能用含糊的 `update`、`fix`、`changes`。
5. 推送前确认当前分支是预期分支；默认不得新建分支。

## 回滚规则

- 首选 `git revert <commit>`，用新提交记录回滚原因，保留历史。
- 单文件恢复可以从备份或旧提交恢复，但恢复结果也必须提交。
- release 附件出错时，应覆盖同一 release 附件并更新 notes，或按用户要求创建新的修正 release；不能静默替换而不更新说明。
- 本地发布包、release 附件、README、CHANGELOG 的版本号和 SHA256 必须一致。
- 回滚完成后必须再次验证：工作树、目标文件、release 附件 digest、仓库私密状态。

## 冻结文件

当前冻结文件：

- `ui/index.html`

`ui/index.html` 是 OCULI / VERTEBRA HTML UI 样例的冻结视觉参考。除非用户明确要求修改这个 UI 参考文件，否则不得修改它的结构、样式、交互、命中区域、可访问状态、favicon、文案或格式。需要提出 UI 改进时，应写在文档里，或另建新原型文件，不得直接改冻结文件。

## 发布规则

- DEV 版本必须继续使用 `DEV` / `dev-...` 标识，不得擅自改成正式版。
- 创建或更新 release 后，必须回查 release 状态、附件名称、附件大小、digest、prerelease 状态和仓库私密状态。
- GitHub 仓库必须保持私密；任何发布前后都要用 GitHub CLI 回查 `visibility=PRIVATE` 和 `isPrivate=true`。

## 备份规则

- 大范围改动、发布前改动、回滚前改动，必须先创建源码备份或确认已有可用备份。
- 备份放在 `_backups/`，不提交到 Git。
- 备份清单必须至少记录创建时间、来源路径、Git HEAD 和当时的 `git status --short`。
